## はじめに

AIコーディングツールの選択肢が増えている。

Cursor、GitHub Copilot、Claude Code、そして最近注目を集めている **Codebuff**。

この記事では、Codebuffを軸に「CLIコーディングエージェントとは何か」を整理し、最終的に**マルチエージェント設計が意味すること**まで掘り下げる。

ツールの比較だけで終わらせたくないので、設計思想まで追いかける。

---

## Codebuffとは

[Codebuff](https://www.codebuff.com/) は、2025年1月にリリースされたCLI型のAIコーディングエージェントだ。

```bash
npm install -g codebuff
cd your-project
codebuff "認証機能を追加して"
```

これだけで動く。IDEのプラグインは不要。ターミナルの中で完結する。

**基本スペック：**

- YC（Y Combinator）出身
- 2025年9月にOSS化（[GitHub: CodebuffAI/codebuff](https://github.com/CodebuffAI/codebuff)）
- GitHub Stars: 3.4k（2026年3月時点）
- TypeScript SDK提供（CI/CDやカスタム統合が可能）
- モデル非依存（OpenRouter経由で任意のモデルに接続可能）
- 料金: 月500クレジット無料、それ以降は従量制

## 他ツールとの比較

### Cursor / GitHub Copilot との違い

| | Codebuff | Cursor | GitHub Copilot |
|---|---|---|---|
| インターフェース | CLI | IDE（VS Code fork） | IDE プラグイン |
| 操作単位 | コードベース全体 | ファイル〜プロジェクト | 行〜関数 |
| 自律性 | 高（テスト実行・反復まで） | 中 | 低 |
| セットアップ | `npm install -g` のみ | IDEインストール必要 | 拡張機能インストール |
| CI/CD統合 | SDK経由で可能 | 難しい | 限定的 |

Cursor は「IDE環境でAIと協力する」イメージ。Codebuff は「AIにプロジェクトを渡してやってもらう」イメージ。

### Claude Code との比較

Anthropicが提供するClaude CodeもCLI型エージェントで、比較されることが多い。

Codebuffは自社の評価指標 **BuffBench**（175以上のOSSリポジトリからの実世界タスク）で以下のスコアを報告している：

- Codebuff: **61%**
- Claude Code: **53%**

速度面では、同一タスクで **Codebuffが平均3倍速** という結果も出ている（6分45秒 vs 19分37秒）。

ただしベンチマークはあくまで参考値。実際のプロジェクトの性質によって結果は変わる。

## Codebuffの核心：マルチエージェント設計

ここからが本題だ。

Codebuffが他のツールと根本的に違うのは、**単一のLLMに全部やらせない** という設計方針だ。

内部では4つのエージェントが連携している：

```
ユーザーの指示
    ↓
┌─────────────────────────────────┐
│  FilePicker                     │
│  コードベースをスキャンし、       │
│  関連ファイルを特定する          │
└────────────────┬────────────────┘
                 ↓
┌─────────────────────────────────┐
│  Planner                        │
│  変更の戦略と順序を設計する      │
└────────────────┬────────────────┘
                 ↓
┌─────────────────────────────────┐
│  Editor                         │
│  実際のコード変更を実行する      │
└────────────────┬────────────────┘
                 ↓
┌─────────────────────────────────┐
│  Reviewer                       │
│  変更内容を検証し、問題があれば  │
│  フィードバックを返す            │
└─────────────────────────────────┘
```

### なぜ分けるのか

一つのLLMに「ファイルを選んで、計画立てて、コード書いて、検証して」を全部頼むと、コンテキストが混在する。

役割を分けることで：

1. **各エージェントが自分の仕事に集中できる** — FilePicker はコードの内容を気にせず「関連性」だけを判断できる
2. **エラーのカスケードを防ぐ** — Reviewer が問題を検出した場合、Editor に差し戻せる
3. **並列処理が可能** — MAX モードでは複数の編集戦略を同時に走らせる（これが3倍速の理由）

## 実際に使ってみる

```bash
# インストール
npm install -g codebuff

# プロジェクトに入って起動
cd your-project
codebuff

# または直接プロンプトを渡す
codebuff "Add TypeScript types to all functions in src/"

# MAX モード（並列戦略、高速・高コスト）
codebuff --max "Refactor the authentication module"

# PLAN モード（変更前に計画を確認できる）
codebuff --plan "Migrate from CommonJS to ESM"
```

**セッション継続も可能：**

```bash
# 前回の会話を続ける
codebuff --continue

# 特定のセッションIDで再開
codebuff --continue 2026-03-04T08-49-41.292Z
```

テスト実行・git操作・反復も自律的に行う。「やってみたら型エラーが出た、直す」というサイクルを自分で回す。

---

## MAXモードとは何か

Codebuffには通常モードの他に **MAX モード** がある。

通常: 一つの戦略でコードを変更
MAX: 複数の戦略を並列で実行し、最も良い結果を採用

```
        ┌── Strategy A ──┐
指示 ──→├── Strategy B ──┤──→ 最善を選択 ──→ 適用
        └── Strategy C ──┘
```

これが速度3倍の正体だ。直列に試行錯誤するのではなく、最初から並列に走らせる。

コスト（API費用）は増えるが、時間は大幅に短縮できる。

## TypeScript SDKで何ができるか

2025年9月のOSS化と同時に、TypeScript SDKも公開された。

```typescript
import { Codebuff } from 'codebuff';

const agent = new Codebuff({
  model: 'claude-3-5-sonnet', // モデルは選べる
  cwd: './my-project',
});

// コードタスクを実行
const result = await agent.run('Add error handling to all API calls');
console.log(result.changes); // 変更されたファイルの一覧
```

これによって：

- **CI/CDパイプラインへの組み込み** — PRがマージされたら自動でリファクタリング走らせる、など
- **カスタムエージェントの構築** — Codebuffのエージェント設計を自分のプロダクトに組み込む
- **テスト自動修正** — テストが落ちたら自動でコードを直す

が可能になる。

## マルチエージェント設計が示すもの

少し立ち止まって考えたい。

Codebuffのマルチエージェント設計は、**「一つの知性が全部やる」より「役割を分けた複数の知性が協力する」方が良い結果を出せる** という仮説に基づいている。

これは、ソフトウェア工学の歴史と同じ方向を向いている。

モノリシックなシステムより、マイクロサービス。一人のスーパーエンジニアより、専門性を持ったチーム。すべてを知っている設計より、関心の分離。

LLMも例外ではなかった、ということだ。

### 知識の限界 vs 役割の分担

一つのLLMに全部やらせると、コンテキストウィンドウという物理的な制約に当たる。大きなコードベースで「ファイルの選択」と「コードの変更」と「検証」を同時にやろうとすると、どこかが疎かになる。

役割を分けることで、各エージェントは自分の担当だけに集中できる。FilePicker は「このファイルが関連するか」だけを考えればいい。Planner は「どう変更するか」だけを考えればいい。

**専門化によって、全体の精度が上がる。**

### 自律性とコントロールのトレードオフ

マルチエージェント設計には、もう一つの側面がある。

各エージェントが自律的に動くということは、人間が介入できるポイントが減るということでもある。

PLAN モードが用意されているのはそのためだ。Planner が計画を立てた段階で人間が確認できる。「この方向でいいですか？」という確認ステップを挟める。

自律性を高めながら、人間のコントロールを失わない設計。これはAIシステム全般に共通する課題で、Codebuffはそれを明示的にモードとして提供している。

## いつCodebuffを使うべきか

使うべき場面：

- **大規模リファクタリング** — 複数ファイルにまたがる変更、型定義の追加、命名規則の統一など
- **反復的な修正** — テストが通るまで自律的に試行させたい
- **CI/CDへの組み込み** — 人間が介さない自動化パイプライン
- **IDE縛りをなくしたい** — リモート開発環境、サーバー上での作業

使わなくていい場面：

- 1〜2行の小さな変更（普通に書いた方が早い）
- コードの意図を深く理解しながら設計する場面（思考プロセスを外注したくない）
- コンテキストを積み上げながら対話的に進めたい場面（Cursorの方が向いている）

## まとめ

Codebuffは「CLIで動くコーディングエージェント」という説明に収まるツールだが、その設計には考えさせられる要素がある。

マルチエージェント構造は、単なる技術的な最適化ではない。**「一つの万能な知性」より「役割を持った複数の知性の協力」の方が良い仕事をする** という考え方の実装だ。

これはソフトウェア設計の歴史が証明してきたことで、LLMの世界でも同じ原理が働いている——Codebuffはそれを示している。

自律性を高めながらコントロールを失わない設計、並列処理による高速化、SDK経由でのパイプライン統合。これらを組み合わせると、Codebuffが目指しているのは「コードを書くツール」ではなく「コーディングというプロセスを再設計するツール」だとわかる。

使ってみる価値がある。そして使いながら、自分がどの部分を手放せて、どの部分を手放したくないかを考えてみると面白いかもしれない。🐾
